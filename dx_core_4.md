# DX Core 4

DX Core 4 は、開発者生産性を測定するための統一的・実践的・バランスの取れたフレームワークです。**DORA・SPACE・DevEx の 3 フレームワークを包含**し、4 つの dimension（スピード / 有効性 / 品質 / インパクト）で生産性を多面的に捉えます。2024 年 12 月に [DX 社](https://getdx.com) の Laura Tacho(CTO)と Abi Noda(CEO)が発表しました。

## 背景

エンジニアリングリーダーから「DORA・SPACE・DevEx の違いは何か、結局自分たちは何を測ればよいのか」という問い合わせが繰り返し寄せられていました。各フレームワークは次のような限界を抱えています。

- **DORA**: 規範的(prescriptive)な指標を提供しますが、システムパフォーマンスに焦点があり対象範囲と用途が限定的です
- **SPACE**: 自分でメトリクスを定義する枠組みですが、実際に運用に落とすのが難しいです
- **DevEx**: 開発者体験と自己申告指標に焦点が当たりますが、生産性のより広い文脈から切り離されています

DX Core 4 はこれらを統合する形で設計され、300 を超える組織で検証されています。報告されている代表的な成果は、エンジニアリング効率 3〜12% 向上、フィーチャー開発に費やす時間 14% 増です。DORA・SPACE・DevEx の研究者である Nicole Forsgren、Margaret-Anne Storey、Thomas Zimmermann がアドバイザーに名を連ねています。

## なぜ 4 つの dimension が必要か

スピードを上げる施策が品質や有効性など他の側面に悪影響を与えることがあるため、1 つの指標を単独で追うと開発者の不安や非生産的な行動を生みやすくなります。複数の dimension を**対抗関係(opposing metrics)** として組み合わせ、バランスを取りながら見ることが本フレームワークの中心思想です。

## 4 つの dimension

|              | スピード                                                | 有効性                                                                          | 品質                                                              | インパクト                                                                                |
| ------------ | ------------------------------------------------------- | ------------------------------------------------------------------------------- | ----------------------------------------------------------------- | ----------------------------------------------------------------------------------------- |
| **主要指標** | エンジニアあたりの Diff 数*<br>(PR または MR)<br><br>*個人レベルでは使用しない | 開発者体験指数 (DXI)<br><br>（DX 社が開発した、主要なエンジニアリング・パフォーマンス要因の指標） | 変更失敗率                                                        | 新機能(Capability)開発に費やす時間の割合                                                  |
| **副次指標** | ・リードタイム<br>・デプロイ頻度<br>・体感的なデリバリー速度  | ・10 件目の PR までの時間<br>・デリバリーのしやすさ<br>・望ましくない離職率*<br><br>*組織レベルのみ | ・デプロイ失敗からの復旧時間<br>・体感的なソフトウェア品質<br>・運用健全性とセキュリティ指標 | ・新たな取組の進捗と ROI<br>・エンジニアあたりの売上*<br>・売上に対する R&D の割合*<br><br>*組織レベルのみ |

---

## ① スピード(Speed)

開発者がどれだけ速く価値を届けられているかを表す dimension です。

- **主要指標: エンジニアあたりの Diff 数(PR / MR)** — Meta・Microsoft・Uber など大手テック企業でも生産性を理解・改善するための重要なインプットとして使われています。ただし利用には注意が必要で、**個人評価の指標として使わない**ことが前提です
- **副次指標**: リードタイム / デプロイ頻度 / 体感的なデリバリー速度

スピード・スループット指標を単独で使うと、開発者の恐怖心や非生産的な行動を招きやすい点が原典で繰り返し警告されています。必ず後述の対抗指標（DXI など）とセットで運用します。

---

## ② 有効性(Effectiveness)

開発者が業務を効果的に進められる環境にあるかを表す dimension です。スピードと対をなし、バランスの取れた生産性の議論を促す位置づけです。

- **主要指標: 開発者体験指数 (DXI)** — DX 社が開発した、エンジニアリング・パフォーマンスを左右する主要な要因を総合する指標です
- **副次指標**: 10 件目の PR までの時間 / デリバリーのしやすさ / 望ましくない離職率（組織レベルのみ）

---

## ③ 品質(Quality)

提供されたソフトウェアがどれだけ信頼でき、健全に動き続けるかを表す dimension です。

- **主要指標: 変更失敗率**
- **副次指標**: デプロイ失敗からの復旧時間 / 体感的なソフトウェア品質 / 運用健全性とセキュリティ指標

---

## ④ インパクト(Impact)

エンジニアリングの労力が事業価値にどれだけ結びついているかを表す dimension です。

- **主要指標: 新機能(Capability)開発に費やす時間の割合**
- **副次指標**: 新たな取組の進捗と ROI / エンジニアあたりの売上（組織レベルのみ）/ 売上に対する R&D の割合（組織レベルのみ）

---

## 運用上の前提

DX Core 4 を健全に機能させるための前提として、原典では次の点が強調されています。

1. **対抗指標でバランスを取る** — スピード単独ではなく、DXI などの体験・有効性指標と組み合わせて見ます
2. **目標や報酬に紐づけない** — 指標自体を達成目標化すると、グッドハートの法則どおり数値合わせが起きます
3. **適切なコミュニケーション** — 誤用・濫用を避けるため、運用の意図と限界をチームに共有します

また「個人レベルでは使用しない」と注記された指標（エンジニアあたりの Diff 数など）は、組織・チームの傾向を見るためのものであり、個人評価・査定への適用は明確に推奨されていません。

---

## データ収集の手法

DX Core 4 は次の 3 種類の計測手法を組み合わせることで、最も包括的な開発者生産性のビューが得られるとされています。

| 手法               | メリット                                       | 課題                                              |
| ------------------ | ---------------------------------------------- | ------------------------------------------------- |
| システムメトリクス | 客観的な指標をリアルタイムで取得できる         | システム横断での可視化とデータの正規化            |
| 自己申告メトリクス | 迅速なデータ収集と体験指標の取得               | 設問設計、回答率                                  |
| 体験サンプリング   | 特定の場面に絞ったリアルタイムの洞察           | セットアップの複雑さ、データ収集にかかる時間      |
