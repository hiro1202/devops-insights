# DevEx

DevEx(Developer Experience)は、開発者が日々の業務にどう関わり、それをどう解釈し、どんな意味を見出すかを捉える考え方です。福利厚生や職場環境を整えることではなく、日々の作業から摩擦を取り除くことに焦点を当てます。

DevEx は、開発者の満足度だけを扱うものではありません。ビルド・テスト・レビュー・デプロイの待ち時間、コードベースの理解しやすさ、会議や通知による中断、ドキュメントやツールの使いやすさなど、開発者の生産性を左右する実務上の条件を扱います。

## 3 つの中核要素

| 要素 | 概要 |
| --- | --- |
| **Feedback Loops** — フィードバックループ | 変更に対してどれだけ早く、質の高いフィードバックを得られるか |
| **Cognitive Load** — 認知負荷 | 開発者が作業を進めるために必要な精神的負荷がどれだけ大きいか |
| **Flow State** — フロー状態 | 中断されずに深く集中して作業できる時間を確保できているか |

---

## Feedback Loops — フィードバックループ

開発者がコードを書き、テストし、反復する「マイクロサイクル」のスピードと質を表す。フィードバックが速いとは、コンパイル・テスト実行・コードレビュー・デプロイがそれぞれ素早く進む状態を指す。

**メトリックの例**

- コンパイル(ビルド)の速度
- テスト実行の速度
- コードレビューの所要時間
- デプロイの所要時間

---

## Cognitive Load — 認知負荷

ソフトウェア開発には本質的な複雑さがある。しかし**不必要な**複雑さは、生産性と満足度を損なう。認知負荷とは、開発者が仕事を完遂するために必要な精神的努力のこと。

**高い認知負荷の発生源**

- 不十分なドキュメントで、仕組みをリバースエンジニアリングしなければならない
- 一貫性のないツールで、類似したタスクごとに異なるアプローチを学ぶ必要がある
- 不明確なプロセスで、試行錯誤を繰り返して「正しい方法」を見つけ出さなければならない
- ワーキングメモリに保持すべき文脈が多すぎる複雑なアーキテクチャ

認知負荷が高いと、開発者は本質的な問題解決ではなく、付随的な作業に精神的エネルギーを費やす。技術的には生産的に見えても、消耗しフラストレーションを溜めることになる。

認知負荷を下げるとは、明確さを生み出すこと。明瞭なドキュメント、一貫したツール、自明なプロセス、そして「コード自身が物語る」よう整理されたコードが必要となる。

---

## Flow State — フロー状態

開発者がよく「ゾーンに入る」と表現する、深く集中した状態。複雑な問題が扱いやすくなり、生産性が大きく上がる。パフォーマンスにも満足度にも不可欠。

**必要な条件**

- 中断されないまとまった時間(最低 2〜4 時間)
- 明確な目標と、明確に定義されたタスク
- 適切な難易度(簡単すぎず、不可能でもない)
- 仕事の進め方に対する自律性

**主な阻害要因**

現代の職場環境は、システム的にフロー状態を壊しがちである。

- 頻繁な会議
- Slack の通知
- 不明確な優先順位
- 頻繁なコンテキストスイッチ

フロー状態を守れている組織では、開発者の生産性・満足度・成果物の品質のすべてが大きく改善する。
